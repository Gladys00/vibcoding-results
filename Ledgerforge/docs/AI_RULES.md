# AI 维护规则

本文档用于约束后续 AI 或 Codex 维护本项目时的行为。

## 总原则

1. 先读代码，再做判断。
2. 不能根据想象补功能说明。
3. 无法从当前代码确认的内容，必须标记为“未知”。
4. 用户要求只讨论或只写文档时，不得修改业务代码。
5. 用户未明确要求提交或推送时，不执行 Git 提交、推送、打标签。

## 当前项目事实

维护时必须牢记：

- 当前项目是静态前端 MVP。
- 当前没有后端。
- 当前没有数据库。
- 当前没有 API。
- 当前没有登录和权限。
- 当前数据保存在 localStorage。
- 当前版本号在代码中为 V1.3.2.2。

## 禁止事项

除非用户明确要求，否则不得：

- 修改 `assets/` 下的业务代码。
- 修改 `index.html`。
- 修改配置文件。
- 重构页面结构。
- 新增后端、数据库或 API。
- 新增依赖管理工具。
- 改动 Git 分支、提交、标签或远程仓库。
- 清空 localStorage 或重置用户数据。

## 文档维护规则

修改功能时，应同步检查以下文档是否需要更新：

- `docs/PROJECT.md`
- `docs/ARCHITECTURE.md`
- `docs/FEATURES.md`
- `docs/CHANGELOG.md`
- `docs/BUG_HISTORY.md`

如果只是探索或讨论，不应提前修改文档。

## 字段与统计规则

维护字段和统计时必须保持一致：

- 同一业务字段在列表、详情、编辑、统计中使用一致名称。
- 台账金额当前按 `数量 * 单价` 计算。
- 不应直接相信导入文件中的金额作为最终统计金额，除非用户确认变更统计口径。
- 欠款金额当前作为校验和提示字段，不直接作为核心入账统计来源。
- 应收按 `收入 - 已收` 计算。
- 应付按 `支出 - 已付` 计算。

## Excel 导入规则

维护导入功能时必须注意：

- 字段映射应优先使用系统维护的映射规则。
- 表格列名不应重复映射到多个系统字段。
- 导入映射页的“调整”应让用户选择已有系统字段。
- 新增系统字段应通过字段配置完成，不应在导入调整中混入新增字段流程。
- Sheet 选择状态应支持全选、多选、取消选中，并在页面切换后保持。

## UI 维护规则

本项目当前 UI 是现代 SaaS 风格。

维护时应遵守：

- 不大幅修改整体布局。
- 不随意移动主导航。
- 不破坏现有页面流程。
- 不增加无必要的新页面。
- 弹窗遮罩点击不关闭。
- 危险操作必须二次确认。
- 用户操作必须有 Toast 反馈。
- 弹窗按钮不得存在空实现。

## 数据兼容规则

`assets/data-store.js` 中存在数据规范化和旧字段兼容逻辑。

维护时应注意：

- 不随意更改 localStorage key。
- 更改数据结构时必须考虑旧数据迁移。
- 删除字段或改名字段前必须确认统计、导入、报表和详情页是否受影响。

## 验证规则

如果修改业务代码，至少应检查：

- 应用可以打开。
- 主要路由可以切换。
- 新增、编辑、删除有反馈。
- 删除有确认。
- Excel 导入流程不报错。
- 台账金额仍按 `数量 * 单价`。
- 统计口径与 Tooltip 文案一致。

当前项目未发现自动化测试配置，因此自动化测试能力为未知。

## Git 规则

用户已经表达过对 Git 工作流仍在熟悉阶段。

因此：

- 推送前必须解释清楚会影响哪个分支。
- 打标签前必须解释标签含义。
- 不应把本地修复误说成已发布远程。
- 不应在用户只要求本地修改或文档时执行 Git 操作。

## 安全规则

- 不要把 token、密钥、账号密码写入代码或文档。
- 不要在文档中记录用户的私密凭据。
- 不要把本地临时文件路径当作正式项目能力。
